# 리뷰 리포트 포맷 표준

모든 리뷰어(architecture/performance/style)는 이 포맷으로 `_workspace/`에 산출물을 기록한다. 일관된 포맷이라야 리드가 종합 리포트로 병합할 수 있다.

## 개별 리뷰어 리포트 구조

```markdown
# {영역} 리뷰 — {YYYY-MM-DD}

## 요약
- 리뷰 대상: {파일 수}개 파일 / {diff | 전체}
- 발견: High {n} · Medium {n} · Low {n}
- (style 전용) flutter analyze: error {n} · warning {n} · info {n}
- 한 줄 총평: {...}

## 발견 항목

### [심각도] 제목
- **위치**: `lib/features/.../foo.dart:42`
- **근거**: 무엇이 왜 문제인가 (Why 중심)
- **영향**: (성능이면 리빌드 빈도/조회 횟수 등 / 그 외 구조·일관성 영향)
- **권장 수정**: 구체적 개선 방향 (코드 스니펫은 핵심만)
- **확신도**: 확실 | 확인 필요 | 잠재적

(발견마다 위 블록 반복. 심각도 높은 순으로 정렬.)
```

## 심각도 공통 정의

| 심각도 | 의미 |
|--------|------|
| **High** | 버그 유발·구조 붕괴·명백한 자원 낭비. 우선 수정 권장. |
| **Medium** | 일관성·유지보수성·잠재 성능 저하. 수정 권장. |
| **Low** | 개선 여지, 선택적. |

## 확신도 표기

- **확실**: 코드를 읽고 근거가 명확.
- **확인 필요**: 단정하려면 추가 맥락 필요(런타임 동작, 의도 등).
- **잠재적**: 정적 분석상 의심되나 실측·맥락 없이는 단정 불가.

추측으로 "확실"을 남발하지 않는다. 근거 파일·라인이 없으면 항목을 만들지 않는다.

## 종합 리포트 구조 (리드가 작성)

```markdown
# 종합 코드 리뷰 — {YYYY-MM-DD}

## 개요
- 대상 범위 / 참여 리뷰어 / 총 발견 수(심각도별)

## 우선순위 Top (영역 교차)
세 영역에서 모인 High 항목 + 교차 발견(한 코드가 여러 영역에서 지적된 것)을 묶어 우선순위로 제시.

## 영역별 발견
### 아키텍처
### 성능
### 코드 스타일
(각 리뷰어 리포트 요약 + 링크)

## 교차 이슈
서로 다른 리뷰어가 같은 파일/원인을 다른 각도로 지적한 항목. 출처 병기.

## 상충/미해결
리뷰어 간 의견이 갈린 항목(삭제하지 않고 양측 출처 병기). 누락된 리뷰(실패한 에이전트) 명시.

## 권장 액션 순서
수정 착수 추천 순서 (영향 대비 비용 기준).
```
